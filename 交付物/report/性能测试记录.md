# 性能测试记录

## 一、测试目标

对比“四代后代查询”在无索引访问方式和有索引访问方式下的执行效率差异，并结合 `EXPLAIN ANALYZE` 执行计划说明索引优化效果。

本次测试使用 PostgreSQL 10 万级成员数据：

| 指标 | 数值 |
|---|---:|
| 族谱数量 | 10 |
| 成员总数 | 100000 |
| 最大单族谱成员数 | 50000 |
| 测试数据库 | `genealogy_db` |
| 测试根成员 | `root_member_id = 1` |
| 测试族谱 | `genealogy_id = 1` |

完整执行计划已保存：

- `交付物/report/explain_no_index.txt`
- `交付物/report/explain_with_index.txt`

## 二、测试 SQL

为贴合应用实际访问模式，本次测试在递归查询中显式限定 `genealogy_id`，避免跨族谱扫描。

```sql
WITH RECURSIVE descendants AS (
    SELECT parent_id, child_id, 1 AS depth
    FROM parent_child_relations
    WHERE genealogy_id = 1
      AND parent_id = 1
    UNION ALL
    SELECT r.parent_id, r.child_id, d.depth + 1
    FROM parent_child_relations r
    JOIN descendants d ON r.parent_id = d.child_id
    WHERE r.genealogy_id = 1
      AND d.depth < 4
)
SELECT d.depth, m.member_id, m.name
FROM descendants d
JOIN members m ON m.member_id = d.child_id
WHERE d.depth = 4;
```

该查询含义：从指定曾祖父节点出发，递归查找 4 层后代，并返回第 4 层成员。

## 三、测试方法

### 3.1 无索引访问方式

为了不破坏数据库中的主键、唯一约束和正式索引，本次没有物理删除索引，而是通过 PostgreSQL 会话参数关闭索引扫描和位图索引扫描，模拟无索引顺序扫描访问方式：

```sql
SET enable_indexscan = off;
SET enable_bitmapscan = off;

EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
WITH RECURSIVE descendants AS (...);
```

### 3.2 有索引访问方式

恢复 PostgreSQL 默认优化器行为，允许使用索引：

```sql
RESET enable_indexscan;
RESET enable_bitmapscan;

EXPLAIN (ANALYZE, BUFFERS, VERBOSE)
WITH RECURSIVE descendants AS (...);
```

相关索引来自 `交付物/sql/indexes.sql`：

```sql
CREATE INDEX IF NOT EXISTS idx_parent_child_parent
    ON parent_child_relations (genealogy_id, parent_id);

CREATE INDEX IF NOT EXISTS idx_parent_child_child
    ON parent_child_relations (genealogy_id, child_id);
```

## 四、测试结果

| 项目 | 无索引访问方式 | 有索引访问方式 |
|---|---:|---:|
| Planning Time | 3.067 ms | 3.028 ms |
| Execution Time | 92.544 ms | 3.292 ms |
| `Seq Scan` 次数 | 3 | 0 |
| `Index Scan` 次数 | 0 | 1 |
| `Hash Join` 次数 | 2 | 0 |
| `Nested Loop` 次数 | 0 | 2 |

性能提升倍数：

```text
92.544 / 3.292 ≈ 28.11 倍
```

---

## 五、执行计划摘要

### 5.1 无索引访问方式

无索引访问方式下，执行计划中出现多次 `Seq Scan`，说明 PostgreSQL 需要扫描较大范围的 `parent_child_relations` 数据，再通过 Hash Join 完成递归扩展。

关键特征：

- `Seq Scan` 次数：3
- `Hash Join` 次数：2
- 没有使用 `Index Scan`
- 总执行时间：`92.544 ms`

### 5.2 有索引访问方式

有索引访问方式下，执行计划不再出现 `Seq Scan`，递归扩展可以根据 `(genealogy_id, parent_id)` 快速定位子节点，再用主键索引访问 `members`。

关键特征：

- `Seq Scan` 次数：0
- `Index Scan` 次数：1
- `Nested Loop` 次数：2
- 总执行时间：`3.292 ms`

---

## 六、结论

针对“四代后代查询”，索引优化效果明显：

- 无索引访问方式依赖顺序扫描，递归层数增加时扫描成本快速上升。
- 建立 `(genealogy_id, parent_id)` 索引后，递归查询可以直接按父节点扩展子节点。
- 在 10 万级成员数据上，本次测试执行时间从 `92.544 ms` 降低到 `3.292 ms`，约提升 `28.11` 倍。

因此，`parent_child_relations(genealogy_id, parent_id)` 是后代查询和树状预览的关键索引；`parent_child_relations(genealogy_id, child_id)` 则是祖先查询的关键索引。
