# 答辩问题准备

> 本文档用于帮助没有参与编码的同学应对老师提问。回答时不要背代码，重点讲清楚设计原因、实现思路和取舍。

---

## 一、项目整体

### 1. 你们这个系统解决什么问题？

系统解决族谱管理问题。它支持用户创建族谱、录入成员、维护亲子和婚姻关系、查询祖先、查询两个人之间的亲缘路径，并支持较大规模模拟数据的导入和性能测试。

### 2. 为什么选择 Flask？

Flask 轻量、开发快，适合课程项目。它可以方便地实现登录、表单提交、模板渲染和路由控制。我们项目重点是数据库设计和查询，因此选择 Flask 可以减少框架复杂度，把精力放在数据库建模、递归查询和索引优化上。

### 3. SQLite 和 PostgreSQL 分别用来做什么？

SQLite 用于本地快速演示，不需要安装数据库服务，克隆后就能启动。PostgreSQL 用于课程交付和大数据测试，因为它支持 `COPY` 批量导入、递归 CTE、触发器、扩展索引和 `EXPLAIN ANALYZE`。

### 4. 为什么不是只用 SQLite？

SQLite 适合演示，但 PostgreSQL 更适合体现数据库课程要求，例如触发器、递归查询、大规模数据导入、索引优化、执行计划分析等。

## 二、数据库设计

### 5. 数据库有哪些核心表？

核心表有 6 张：

- `users`：系统用户；
- `genealogies`：族谱；
- `genealogy_collaborators`：协作者关系；
- `members`：成员；
- `parent_child_relations`：亲子关系；
- `marriages`：婚姻关系。

### 6. 为什么要单独设计 `parent_child_relations` 表？

因为亲子关系是独立关系，不适合直接塞到成员表里。一个成员可能作为父母出现在多个亲子关系中，也可能作为子女有父亲和母亲。拆成关系表后更规范，也方便递归查询祖先和后代。

### 7. 为什么要单独设计 `marriages` 表？

婚姻关系是成员之间的关系，而不是成员本身的属性。单独建表可以表示配偶双方，也方便查询某个成员的配偶，并且避免在成员表中增加不稳定的配偶字段。

### 8. 为什么需要 `genealogy_collaborators` 表？

因为用户和族谱之间是多对多关系。一个用户可以参与多个族谱，一个族谱也可以邀请多个用户协作，所以需要中间表。

### 9. 这个数据库满足第几范式？

整体满足第三范式。每张表只表达一种实体或关系，非主属性依赖主键，不存在明显的传递依赖。例如成员信息放在 `members`，亲子关系放在 `parent_child_relations`，婚姻关系放在 `marriages`，没有把关系信息重复塞到成员表中。

### 10. 如果老师问 BCNF，怎么回答？

可以回答：核心表设计接近 BCNF，因为大多数非平凡函数依赖都由候选键决定，例如用户由 `user_id` 或唯一 `username` 决定，成员由 `member_id` 决定，族谱由 `genealogy_id` 决定。协作关系和亲子关系也通过唯一约束避免重复。报告中我们主要按第三范式进行说明，更稳妥。

### 11. 数据库中有哪些约束？

有主键、外键、唯一约束、CHECK 约束和触发器。比如用户名唯一，成员性别只能是 `M` 或 `F`，死亡年份不能早于出生年份，亲子关系不能自己指向自己，婚姻关系不能自己和自己结婚。

### 12. 为什么要用触发器？

有些约束需要跨表检查，普通 CHECK 做不到。例如父亲必须是男性、母亲必须是女性、父母和子女必须属于同一个族谱、父母出生年份必须早于子女。这些规则需要查询 `members` 表，所以使用触发器实现。

## 三、核心查询

### 13. 什么是递归 CTE？

递归 CTE 是 SQL 中的递归查询写法，语法是 `WITH RECURSIVE`。它适合处理祖先、后代、组织结构这种层级深度不确定的数据。

### 14. 祖先查询怎么实现？

先查当前成员的父母，然后把父母作为新的子节点继续向上查找，直到没有更上层父母为止。SQL 用 `WITH RECURSIVE ancestor_tree AS (...)` 实现。

### 15. 树状族谱怎么实现？

后端使用递归 CTE 查出某个根成员下面指定层数的子孙，再把查询结果组装成树形结构传给模板。前端用树状 CSS 布局展示，夫妻左右并列，子女向下展开。

### 16. 为什么树状展示要限制最大层数？

因为 10 万级数据下，如果一次渲染整棵树，浏览器和后端都可能卡顿甚至内存溢出。限制层数可以保证演示稳定，同时仍然能展示树状关系。

### 17. 亲缘路径查询怎么实现？

把成员看成图上的节点，把亲子关系和婚姻关系看成边。然后从起点成员开始做 BFS，找到终点成员后回溯路径，得到两个人之间的亲缘链路。

### 18. 为什么亲缘路径用 BFS？

BFS 适合在无权图中寻找最短路径。亲子关系和婚姻关系可以看成没有权重的边，所以 BFS 可以找到两人之间较短的关系链。

### 19. 为什么亲缘路径没有用 SQL 实现？

SQL 递归也可以做路径查询，但路径中同时包含亲子和婚姻两类边，应用层 BFS 更容易控制关系标签和路径回溯。课程重点 SQL 查询已经通过 `queries.sql` 实现，亲缘路径作为应用功能放在服务层实现。

## 四、数据生成与导入

### 20. 模拟数据怎么生成？

使用 Python 脚本生成。脚本按照族谱、代际、成员数量生成成员，再生成亲子关系和婚姻关系，最后输出 CSV 文件。

### 21. 是否满足 PPT 的数据规模要求？

满足。交付数据包含 10 个族谱，总成员数 100000，其中最大单族谱 50000 人，每个族谱至少 30 代。

### 22. 每个成员是否至少和另一个成员有亲缘关系？

生成脚本会为相邻代成员建立亲子关系，并为同代成员建立婚姻关系，因此成员会参与亲子或婚姻关系，满足亲缘连接要求。

### 23. 为什么用 CSV 和 COPY 导入？

因为 10 万级数据如果逐条 INSERT 会比较慢。PostgreSQL 的 `COPY` 是批量导入工具，适合一次性导入大量 CSV 数据，效率更高。

### 24. 导入顺序为什么重要？

因为有外键约束。必须先导入 `users`，再导入 `genealogies`，再导入 `members`，最后导入依赖成员的亲子关系和婚姻关系。

## 五、索引和性能

### 25. 你们设计了哪些索引？

主要有：

- `members(genealogy_id, name)`：优化族谱内姓名查询；
- `members(genealogy_id, generation_no)`：优化代际查询；
- `parent_child_relations(genealogy_id, parent_id)`：优化查子女和后代；
- `parent_child_relations(genealogy_id, child_id)`：优化查父母和祖先；
- `marriages(genealogy_id, spouse1_id)` 和 `marriages(genealogy_id, spouse2_id)`：优化配偶查询；
- `members` 姓名的 `pg_trgm` GIN 索引：优化模糊搜索。

### 26. 为什么索引里经常带 `genealogy_id`？

因为系统中有多个族谱，查询通常发生在某一个族谱内。把 `genealogy_id` 放在组合索引前面，可以先缩小数据范围，再查姓名、父节点、子节点或配偶。

### 27. 四代后代查询为什么需要索引？

四代后代查询会不断根据父节点找子节点。如果没有 `parent_id` 相关索引，每一层递归都可能扫描大量关系表。建立 `(genealogy_id, parent_id)` 索引后，可以快速定位某个父节点的子女。

### 28. 性能测试结果是什么？

在 10 万级成员数据上，四代后代查询无索引访问方式约 `92.544 ms`，有索引访问方式约 `3.292 ms`，提升约 `28.11` 倍。

### 29. 为什么说“无索引访问方式”而不是直接删索引？

因为数据库里有主键、唯一约束和正式索引，直接删除会破坏当前演示环境。我们通过 PostgreSQL 会话参数关闭 `indexscan` 和 `bitmapscan` 来模拟顺序扫描，这样能对比索引效果，又不会破坏数据库结构。

## 六、界面与工程

### 30. 为什么成员选择不用全量下拉框？

因为大数据下成员可能有 10 万个，如果一次性加载进下拉框，浏览器会卡顿甚至 OOM。现在采用输入姓名搜索，每次只返回少量候选。

### 31. 为什么成员列表分页？

同样是为了避免一次渲染过多数据。每页显示固定数量成员，用户通过翻页查看，数据库和浏览器压力都更小。

### 32. 项目如何一键启动？

执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\start_app.ps1
```

默认使用 SQLite 演示数据库。若使用 PostgreSQL，需要传入 `DatabaseUrl`。

### 33. 默认账号是什么？

```text
用户名：demo
密码：demo123
```

### 34. 如果页面没有数据怎么办？

执行：

```powershell
python seed_demo_data.py
```

然后重启应用。

## 七、项目不足

### 35. 当前项目还有哪些不足？

可以回答：

第一，祖先查询目前主要是表格展示，后续可以改成树状图。

第二，亲缘路径 BFS 在应用层实现，未来可以进一步研究数据库递归路径查询。

第三，导出某分支目前提供 SQL 示例，未来可以做成页面按钮。

第四，系统面向课程演示，没有做生产级安全审计和权限细粒度控制。

### 36. 如果让你继续优化，你会做什么？

优先做三件事：

- 把祖先查询改成树状图；
- 增加分支导出页面按钮；
- 对亲缘路径查询做按需邻接查询，避免大族谱下一次加载过多关系。

## 八、答辩时的表达建议

如果老师问实现细节，先讲“为什么这样设计”，再讲“在哪个文件实现”。

例如：

“我们把亲子关系拆成单独表，是因为亲子关系是成员之间的多关系，不适合直接作为成员属性。实现上对应 `parent_child_relations` 表，递归查询在 `services.py` 中完成。”

如果不确定某个代码细节，不要硬背，可以回答：

“这个部分主要由 Flask 路由调用 SQLAlchemy 查询完成，具体函数在 `genealogy_app/routes.py` 或 `services.py` 中。设计思路是……”
